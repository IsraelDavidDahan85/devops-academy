Final Project — Make a Linux Box Yours

Provision a fresh Linux machine, customize it end-to-end, and document what's happening underneath. The capstone that ties together every lab in this course.

Course01 — Linux Fundamentals
Time3–5 hours (spread across a few sittings)
DeliverableA Git repo (or .tar.gz) containing: README.md, boot.md, system-report.txt, users.md, storage.md, dotfiles/, bin/, systemd/
FormatPush to GitHub (preferred) — submit the repo URL — or archive and submit as a file.

Capability demonstrated

You can sit down at a fresh, unfamiliar Linux box and within an hour have it set up the way you work — with a prompt you can read, aliases that match your habits, a working $PATH that includes your own scripts, and a clear-eyed understanding of what the machine actually did between power-on and login.

This is the difference between "I can run commands" and "this machine is mine."

The brief

You're handed a fresh Linux VM. It boots straight to a default shell with default colors, no aliases, no custom scripts, no story. Your job: make it yours, and document both what you did and what the machine looked like underneath.

This is exactly what every engineer does on a new laptop or every time they provision a server. Practising it deliberately, once, makes the rest of your career on Linux faster.

Requirements

1 · Provision a real Linux machine

Any of these works:

  • A local VM (multipass, UTM, VirtualBox, Lima, VMware) running Ubuntu, Debian, Fedora, Arch, or similar — anything with systemd.
  • A cloud VM (DigitalOcean droplet, AWS EC2, GCP Compute, Linode, Hetzner — even the free tier).
  • WSL2 on Windows (works, but the boot section will be unusual — note that in boot.md).

It must be fresh. Don't reuse your existing dotfiles or your daily-driver machine.

2 · Customize the shell environment

In your shell's rc file (~/.bashrc or ~/.zshrc), set up:

  • At least 3 aliases that you'd actually use day-to-day. Examples: ll='ls -lah', gs='git status', ..='cd ..'. No copying generic lists — pick ones that match how you think.
  • A custom PS1 that shows at least username, hostname, and working directory. Bonus: add color, add the current git branch.
  • An updated $PATH that includes ~/bin (at the front, so your scripts win over system commands of the same name).

Aliases / PS1 / PATH must survive a new shell — the test is: close the terminal, reopen it, alias still works without re-running anything.

3 · Write at least one script in ~/bin

Any script you'll actually use. Examples:

  • up — runs apt update && apt upgrade (or your distro equivalent) with sudo.
  • weather — curl wttr.in/Tel-Aviv or similar.
  • backup-notes — rsyncs ~/notes to a backup dir with a timestamp.
  • greet — the lab-02 script, kept.

The script must be executable (chmod +x), have a shebang (#!/usr/bin/env bash), and run from any directory (proving your $PATH setup works).

4 · Write boot.md — your machine's boot story

Document the actual boot sequence as observed on YOUR VM. Use lab-03's template as a starting point, but every value must be from your machine.

Cover at minimum:

  • Firmware/bootloader your distro uses (BIOS/UEFI, GRUB/systemd-boot).
  • Kernel version (from uname -r) and the first 5 lines of dmesg.
  • The default systemd target on your box, plus 3 services that started during boot (use systemctl list-units --type=service).
  • One example for each of the 7 file types (-, d, l, c, b, p, s) — full ls -l output.
  • Three FHS directories you actually opened, with one observation per directory ("/etc on this box has 142 files. The most interesting one was…").

5 · Build system-report.txt with pipes

One file. Generated by at least three different compositions using pipes, redirection, or globs. Example:

{
  echo "=== top 10 largest files in /var/log ==="
  ls -lS /var/log 2>/dev/null | head -n 11

  echo "=== number of installed packages ==="
  dpkg -l 2>/dev/null | grep -c "^ii" || rpm -qa 2>/dev/null | wc -l

  echo "=== users with login shells ==="
  cat /etc/passwd | grep -v "/nologin\|/false" | cut -d: -f1
} > system-report.txt

The point isn't the exact pipelines — it's that you composed them yourself and the file proves you understand shell plumbing.

6 · Add a non-root user with a shared group

The box you got booted as root or as a default cloud user. Don't operate as root long-term. Create a real user for yourself plus a shared group, and use the right permissions on a shared directory:

  • Create a new user (your name) with a home dir and a real shell.
  • Add them to the sudo group (or wheel on RHEL) so they can run privileged commands.
  • Create a devs group, add yourself, and make /srv/shared owned by root:devs with mode 2775 (SGID — see Lab 04).
  • Verify with id, getent, and ls -ld /srv/shared.

Document the exact commands in users.md, including a copy of the /etc/passwd and /etc/group rows for the user/group you added.

7 · Mount an extra volume

Add storage beyond the root filesystem and make it survive a reboot:

  • Either attach a second disk (cloud / VM-host UI) or create a loopback device from a regular file (see Lab 05).
  • Format it with ext4 (or xfs), label it, mount it at /data.
  • Persist the mount in /etc/fstab by UUID (never device path) with nofail.
  • Reboot. Verify the mount survives — check with findmnt /data.

Document the layout (lsblk output before and after) and your /etc/fstab line in storage.md.

8 · Deploy a custom systemd service

Replace the cron-jobby instinct with a real .service + .timer pair (see Lab 05):

  • Write a small script under /usr/local/bin/ that does something useful — appends a heartbeat to /data/heartbeat.log, snapshots df -h, or rotates your system report.
  • Save a .service file under /etc/systemd/system/ that runs the script as your non-root user.
  • Save a paired .timer file that fires it on a schedule (every 5 min is fine for the demo).
  • systemctl daemon-reload, enable --now the timer, then check systemctl list-timers and journalctl -u <svc> -f.

Commit both unit files to your repo under systemd/.

9 · Submit as a Git repo

Initialize Git in your VM. Put the artifacts under a sensible structure:

my-linux-final/
├── README.md              # 1-page intro: what's here, how to read it
├── boot.md                # your boot narrative
├── users.md               # how you set up the user, group, /srv/shared
├── storage.md             # disk layout + your fstab line
├── system-report.txt      # pipe-built system snapshot
├── dotfiles/
│   ├── bashrc-snippet     # the lines you added to ~/.bashrc
│   └── ps1-example.txt    # screenshot or copy of your prompt
├── bin/
│   └── greet              # (or your other script)
└── systemd/
    ├── heartbeat.service  # your service unit
    └── heartbeat.timer    # paired timer unit

Push to GitHub and submit the repo URL.

Success criteria

You pass when a reviewer can:

  1. Read your README.md and understand the scope of what you did in under two minutes.
  2. Verify your aliases are real (copy them into their own shell — they work).
  3. Read boot.md and find concrete, machine-specific values (not template placeholders) for each of the boot stages, file types, and FHS observations.
  4. Read system-report.txt and tell which pipe / redirect / glob produced each section.
  5. Look at your bin/ script, run shellcheck on it, and see no critical warnings.
  6. Read users.md and verify you created a real non-root user with sudo + a shared /srv/shared directory using SGID.
  7. Read storage.md, see a UUID-based fstab line, and confirm the extra volume mounts at /data after a reboot.
  8. Copy the unit files from your systemd/ directory onto another machine, daemon-reload, enable the timer, and see it run end-to-end via journalctl.

What we're not grading

  • Aesthetic of your prompt. Make it work for you.
  • Which distro you chose.
  • How many aliases beyond the required three — three real ones beat thirty copy-pasted ones.
  • Whether you used bash or zsh.

Stretch (optional, but encouraged)

  • Two more dotfiles: add ~/.gitconfig with your name + email, and ~/.tmux.conf if you use tmux.
  • Install script: write setup.sh in your repo that, when run on a fresh VM, sets everything up by symlinking your dotfiles and dropping scripts into ~/bin. Idempotent.
  • Service health one-liner: add to system-report.txt the count of failed systemd units (systemctl --failed), and explain in boot.md what each failure means on your box.
  • Public the repo: make it public. The next student in the program might learn from your setup.
