Linux Fundamentals — Guide

A long-form companion to the slide deck. Where the slides give you the picture and the cheat sheet gives you the lookup, this document walks you through every concept in prose — the why and the when, not just the what. Read it linearly the first time; come back to specific sections after a lab.

How this is organised — sections mirror the slide deck one-to-one. Each part of the deck is a part of this guide; each section within a part maps to a slide page (or a small cluster of related pages). Slide page numbers are noted in parentheses.

Why Linux (slide 2)

Linux runs the modern world's plumbing. Every major cloud provider's compute fleet, the vast majority of internet servers, every CI runner, every Android phone, every Raspberry Pi — all Linux. As a DevOps engineer you will spend more hours staring at Linux terminals than at any other interface in your career.

The good news: once you can navigate one Linux machine, you can navigate them all. The conventions are remarkably stable. The same cd, ls, and cat work on a tiny embedded box and on a $50k cloud GPU instance. The shell you learn this week is the shell you'll use a decade from now. The first hour of pain pays off for the next ten years.

You don't need to memorise everything. You need to recognise the categories — "this is a process control thing", "this is a filesystem thing", "this is a network thing" — and know which command family to reach for. The rest is man.

A short history (slide 3)

Linux didn't appear from nowhere. In 1969 Ken Thompson and Dennis Ritchie built Unix at Bell Labs — a small, composable operating system organised around a single idea: small tools that do one thing well, piped together. That philosophy is why grep | sort | uniq | head still works fifty years later. Pipes outlived disco.

In 1991 Linus Torvalds posted his hobby kernel to comp.os.minix ("I'm doing a (free) operating system — just a hobby, won't be big and professional like GNU"). He was wrong about scale; it ate the world. The combination of Linus's kernel and the GNU project's userspace tools became GNU/Linux — commonly just "Linux" — and turned the bazaar model of open-source development from curiosity into industry default.

Git arrived in 2005 — also from Linus, built to manage Linux's own source tree. Docker in 2013 wrapped Linux namespaces and cgroups into developer-friendly containers. Every time you type kubectl apply, you're standing on this lineage.

Part 01 — The Shell (slide 4)

The shell is how you talk to Linux. It's a program that reads what you type, runs it, and shows you the output — a loop you'll be inside for thousands of hours. Modern shells (bash, zsh) are also small programming languages, but for the next few sections we treat the shell as a tool: types, lists, navigates, asks for help. The advanced features come in Part 02.

Your first commands (slide 5)

Every shell session starts with a question — "where am I?" — that pwd ("print working directory") answers. Once you know where you are, ls shows you what's here, and cd <path> moves you. Three commands buy you the entire filesystem.

$ pwd
/home/student

$ ls
Documents  Downloads  notes.md  projects

$ cd projects && pwd
/home/student/projects

A few shortcuts are worth burning in immediately: cd ~ takes you home, cd / goes to the root of the whole filesystem, and cd - jumps back to the last directory you were in (great for ping-ponging between two places).

Paths matter. A path starting with / is absolute — it always means the same thing no matter where you're standing. Anything else is relative to where you are right now. ./foo means "foo in the current directory", ../foo means "foo one level up", and bare foo usually means the same as ./foo.

Make, move, remove (slide 6)

The file-operation primitives are five verbs you'll use every day: mkdir, touch, cp, mv, rm. The flags matter as much as the verbs.

mkdir -p devops/linux/lab-01   # create the path, parents and all
touch notes.md                  # empty file, or update its mtime
cp -r src/ backup/              # copy a directory (-r recursive)
cp -p config.toml /etc/         # copy + preserve mode and timestamps
ln -s /opt/myapp/current latest # create a symlink named "latest"
mv old.md new.md                # rename
mv *.log archive/               # move many files at once
rm -r old/                      # remove recursively
rm -rf old/                     # …force, no prompts. dangerous.

cp -r's recursive flag is the one to internalise first — without it, copying a directory fails. cp -p is for when you actually care about preserving timestamps and permissions (deployment scripts care). ln -s creates a symlink: a pointer to another path. Symlinks are everywhere on a real Linux system — half of /usr/bin is symlinks to alternates.

The cardinal sin. rm -rf doesn't ask twice. Before you press Enter on any rm command that uses a variable ($DIR) or a glob (*), run ls on the same path first. The number of careers nearly ended by an unexpanded variable in rm -rf "$BASE/$SUB" when $BASE turned out to be empty is uncountable.

Hunt with find (slide 7)

Where ls only lists what's in the current directory, find walks the entire tree and filters as it goes. It is the right tool whenever you need to ask the filesystem a question more interesting than "what's here?"

find . -name "*.log"            # every .log file under the current dir
find /home -type f -size +100M  # regular files bigger than 100 MB
find /etc -name "*.conf" -mtime -7   # config files modified in the last week
find . -name "*.tmp" -delete    # …or -exec rm {} \; for more general actions

The argument order matters: first the where (a path), then a chain of filters (-type, -name, -size, -mtime), then optionally an action (-delete, -exec). Quote any pattern with shell metacharacters — without quotes, the shell expands *.log before find ever sees it, which is almost never what you want.

find is recursive by default. To limit depth, add -maxdepth 1. Combined with -exec, it becomes the universal "do X on every match" hammer.

Four ways to read a file (slide 8)

Reading a file sounds simple — but the right tool depends on the file's size and what you want to do with it.

cat short.txt              # dump the whole thing to stdout
less /var/log/syslog       # page through; / search, n/N next, q quit
head -n 20 big.log         # the first 20 lines
tail -n 50 big.log         # the last 50 lines
tail -f /var/log/nginx/access.log   # follow new appends LIVE

cat is for small files — dumping a 10 GB log to your terminal will hurt. less is the safe default for unknown sizes: it loads pages on demand and supports searching. head and tail are for one end of the file; tail -f is the one you'll memorise without trying — it follows a file as new lines are appended, the canonical "watch a log" command. Combined with a service restart in another window, tail -f is half of debugging.

Reading ls -l (slide 9)

The long-format ls -l output looks dense but encodes seven distinct pieces of information per file. Once you can read it, you can read any Unix system on Earth.

-rw-r--r--  1  student  staff  4096  Jan 12 10:14  README.md
└──┬───┘    │  └──┬──┘  └─┬─┘  └─┬─┘  └─────┬─────┘  └────┬────┘
  perms   links  owner  group  size       modified      name

The leftmost 10 characters break down as 1 type + 3 owner + 3 group + 3 other. The type character tells you whether it's a regular file (-), directory (d), symlink (l), or something more exotic (we'll meet the others in Part 03). The three permission triplets each show read, write, and execute — or a - if absent.

Flags stack: ls -lah = long, hidden, human-readable sizes. ls -ltSr = long, sorted by time, then by size, reversed. The full flag glossary is in the cheat sheet; the habit of stacking them is the thing to internalise.

The help system (slide 10)

The shell ships with answers. Before you reach for a search engine, reach for these:

man cp              # the full manual page — flags, examples, see-also
man 5 passwd        # section 5 = file formats; section 1 = commands; 8 = admin
man hier            # the canonical tour of the filesystem hierarchy
cp --help           # quick reference card baked into most commands
whatis ls           # one-line description
apropos compress    # search the manual database for a topic
which python3       # which executable runs when you type the name
type ll             # is it a binary, alias, function, or builtin?

Man pages have sections: section 1 is user commands, section 5 is file formats, section 8 is system administration. When two things have the same name — a command and a config file — you disambiguate with the number: man passwd opens the command, man 5 passwd opens the file-format page describing /etc/passwd.

Memorising man hier is a high-leverage move. Run it once. It's a single page that names every top-level directory in the FHS and tells you what's supposed to live there — the canonical reference, on every Linux machine, no internet required.
