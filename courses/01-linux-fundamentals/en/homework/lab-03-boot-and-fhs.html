Lab 03 — Boot & FHS

Look under the hood — trace the boot timeline with dmesg and journalctl, inspect systemd targets and services, identify all seven Linux file types, and walk the FHS top-level directories.

Course01 — Linux Fundamentals
Time~60 minutes
Deliverable~/lab-03/transcript.txt + ~/lab-03/boot-timeline.md + a one-paragraph reflection
Maps to slides20–22 (Boot process · systemd targets · File types & FHS)
macOS heads-up — this lab requires a real Linux system. Most commands here (journalctl, systemctl, /proc) don't exist on macOS. Use a VM, WSL2, or SSH into a cloud Linux box.

Goal

Stop seeing the operating system as a black box. By the end of this lab you should be able to: trace what happens between power-on and login shell; identify what's running and why; tell a regular file from a device, socket, or named pipe at a glance; and navigate any unfamiliar Linux filesystem by purpose rather than memory.

Setup

A real Linux environment with systemd (Ubuntu, Debian, Fedora, Arch — almost everything modern). Confirm:

uname -a               # kernel + arch
cat /etc/os-release    # distro identity
systemctl --version    # confirm systemd is the init system

Create a workspace for this lab:

mkdir -p ~/lab-03
cd ~/lab-03

Tasks

01The kernel's own logs

dmesg prints the kernel ring buffer — what the kernel itself logged from power-on. On some distros it needs sudo.

dmesg | head -n 20          # the very first lines after boot
dmesg | tail -n 20          # most recent kernel events
dmesg | grep -i "usb"       # what USB devices the kernel saw
dmesg | grep -i "ext4\|xfs\|btrfs"   # filesystem mounts
What's the FIRST thing your kernel logged after boot? What CPU did it detect?

02The journal from this boot

journalctl is systemd's unified log. -b = current boot, -b -1 = previous boot.

journalctl -b | head -n 30         # boot from the start
journalctl -b --no-pager | wc -l   # how many log lines this boot
journalctl -b -p err               # only errors
journalctl --list-boots            # every recorded boot
journalctl _PID=1 -b | head -n 20  # only what PID 1 (systemd) said

03systemd targets and services

Targets are named groups of services. Services are units that systemd manages.

systemctl get-default                       # multi-user.target? graphical.target?
systemctl list-units --type=target          # all active targets
systemctl list-units --type=service --state=running | head -n 20
systemctl list-units --failed               # anything broken?
systemctl status                             # the big-picture tree

Pick one service that's running on your box (e.g. ssh.service, cron.service, NetworkManager.service) and inspect it:

systemctl status <service>.service
systemctl cat <service>.service              # the unit file itself
journalctl -u <service>.service -n 30        # its logs

04Identify all seven file types

The first character of ls -l tells you what kind of file you're looking at: - regular, d dir, l symlink, c char device, b block device, p named pipe, s socket.

Find one example of each:

ls -l /etc/hosts                       # -    regular file
ls -ld /var/log                        # d    directory
ls -l /bin                             # l    symlink (on many distros /bin → /usr/bin)
ls -l /dev/tty                         # c    character device
ls -l /dev/sda 2>/dev/null || ls -l /dev/disk0   # b    block device
ls -l /dev/log 2>/dev/null || find /run -type p 2>/dev/null | head -n1 | xargs -I{} ls -l {}
ls -l /var/run/docker.sock 2>/dev/null || find /run -type s 2>/dev/null | head -n1 | xargs -I{} ls -l {}
Record one example of each type — paste them into a section of boot-timeline.md later.

05FHS — a walking tour

Visit each top-level directory and look at one telling thing.

ls /                       # the whole top
ls /bin | wc -l            # how many user binaries
ls /sbin | head            # system admin binaries
cat /etc/os-release        # /etc = configuration
ls /home                   # one folder per user
ls /var/log | head         # /var = variable data (mostly logs)
ls /usr/share | head       # /usr = installed software, read-only
ls /tmp                    # /tmp = scratch (wiped on reboot, usually)
cat /proc/cpuinfo | head   # /proc = live kernel view
cat /proc/meminfo | head
ls /sys/class | head       # /sys = kernel objects (devices, modules)
ls /dev | head             # /dev = device nodes

06Peek at PID 1

Every Linux process descends from PID 1. On systemd boxes, that's systemd itself.

ps -p 1 -o pid,comm,args
ls -la /proc/1/exe         # symlink to the running binary
cat /proc/1/cmdline | tr '\0' ' '; echo  # how it was invoked
ls /proc/1/fd | head       # file descriptors PID 1 has open

Hints

  • journalctl -f follows new log entries live — like tail -f for the system journal. Useful for watching what happens when you start/stop services in another terminal.
  • Don't isolate a target casually. systemctl isolate rescue.target kicks every other service offline — fine on a personal VM, bad over SSH to a server you care about.
  • /proc is fake. Files there aren't on disk — they're a live view into kernel data. That's why cat /proc/cpuinfo always shows current info.
  • The l file type is everywhere. Many distros have /bin, /sbin, /lib as symlinks into /usr. The "usrmerge" world.
  • If a command needs sudo and you don't have it, note that in your transcript ("dmesg requires sudo on Ubuntu >= 23"). The lab still counts.

Stretch (optional)

  • Service dependency tree: run systemctl list-dependencies multi-user.target (or whatever your default target is). What's at the root?
  • Boot time: systemd-analyze tells you how long boot took. systemd-analyze blame ranks services by startup time. Which service is slowest on your box?
  • Bind mounts in /proc: compare cat /proc/self/mounts with mount output. What's different?
  • Write your own systemd unit (don't enable it). Drop a .service file in ~/.config/systemd/user/ that runs echo hello >> ~/lab-03/hello.log. Run it with systemctl --user start <name>.service. Read its journal.
