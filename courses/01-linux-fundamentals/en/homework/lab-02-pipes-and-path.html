Lab 02 — Pipes & PATH

Compose commands with pipes, redirection, and globbing — then control where the shell finds programs by editing $PATH.

Course01 — Linux Fundamentals
Time~45 minutes
Deliverable~/lab-02/transcript.txt + ~/bin/greet (a custom script on your PATH) + a one-paragraph reflection
Maps to slides14–16 (Pipes & globs · PATH · Prompt)

Goal

Stop treating commands as isolated tools. Learn to compose — pipe the output of one into the input of another, redirect streams to files, and use wildcards to operate on many files at once. Then take control of your environment by editing $PATH so the shell finds your own scripts.

Setup

A Linux terminal (VM, WSL, cloud, or macOS — most flags match). Confirm you're at home:

cd ~
pwd

Create a workspace for this lab:

mkdir -p ~/lab-02/{logs,reports}
cd ~/lab-02

Tasks

01Inspect $PATH

Before you change $PATH, look at what's already there.

echo $PATH
echo $PATH | tr ':' '\n'      # one directory per line
which ls
which python3 || echo "no python3 on PATH"
type cd                       # builtin? alias? file?
What's the FIRST directory on your $PATH? When you type ls, why does /usr/bin/ls run instead of /bin/ls?

02Globbing — wildcards

Create some files to play with:

cd ~/lab-02
touch alpha.txt beta.txt gamma.txt readme.md
touch logs/{2024-01,2024-02,2024-03}.log

Try each wildcard:

ls *.txt                # all .txt files
ls *.md                 # all .md files
ls logs/*.log           # all .log files in logs/
ls logs/2024-0?.log     # ? = exactly one character
ls [ab]*.txt            # starts with a or b
ls [!a]*.txt            # does NOT start with a

03Redirection — > · >> · 2>

Streams: stdout (1), stderr (2). Direct them independently.

ls -la           >  reports/listing.txt    # stdout overwrite
date             >> reports/listing.txt    # stdout append
ls /nope         2> reports/errors.txt     # stderr to file
ls / /nope       >  reports/mixed.txt 2>&1  # both into one file
cat reports/listing.txt
What's in reports/errors.txt? What's the difference between 2> and >?

04Pipes — compose commands

The pipe (|) sends stdout of the left command as stdin to the right. Read each example and run it:

# count how many .log files in /var/log
ls /var/log/*.log 2>/dev/null | wc -l

# the 5 largest files in /etc
ls -lS /etc 2>/dev/null | head -n 6

# everyone running bash right now
ps aux | grep bash | grep -v grep

# unique shells used on this system
cat /etc/passwd | cut -d: -f7 | sort -u

05Build your own pipeline

Combine pipes + globs + redirection. Save the output to a file.

# count error lines across the lab's logs
echo "error_1" >  logs/2024-01.log
echo "info"    >> logs/2024-01.log
echo "error_2" >  logs/2024-02.log
echo "ERROR_3" >  logs/2024-03.log

cat logs/*.log | grep -i "error" | sort -u > reports/all-errors.txt
cat reports/all-errors.txt
wc -l reports/all-errors.txt

06Make your own command, put it on $PATH

Create a tiny script that greets by name.

mkdir -p ~/bin
cat > ~/bin/greet <<'EOF'
#!/usr/bin/env bash
name="${1:-friend}"
echo "Hello, $name. Welcome to lab-02."
EOF
chmod +x ~/bin/greet

Try running it. It won't work yet:

greet world
# bash: greet: command not found

That's because ~/bin isn't on $PATH. Add it for this shell:

export PATH="$HOME/bin:$PATH"
echo $PATH | head -c 60        # confirm ~/bin is first
greet world                    # → Hello, world. ...

07Persist the change

export only lasts for the current shell. To make it permanent:

# bash users:
echo 'export PATH="$HOME/bin:$PATH"' >> ~/.bashrc

# zsh users:
echo 'export PATH="$HOME/bin:$PATH"' >> ~/.zshrc

# reload without logging out:
source ~/.bashrc   # or source ~/.zshrc

Open a new terminal and type greet $(whoami) — it should work without re-exporting.

08Capture your work

cd ~/lab-02
history | tail -n 100 > transcript.txt
ls -laR ~/lab-02 ~/bin/greet >> transcript.txt
echo "--- PATH ---"          >> transcript.txt
echo $PATH                    >> transcript.txt

Hints

  • 2>/dev/null silences stderr — useful when a command spams "Permission denied" for files you don't care about.
  • &> in bash redirects both stdout and stderr at once. Less typing than > file 2>&1.
  • type <cmd> tells you if a command is a binary on PATH, an alias, a shell builtin, or a function. Reach for it when "where does this come from?" matters.
  • Shell rc files: bash reads ~/.bashrc for interactive shells, ~/.bash_profile for login shells; zsh reads ~/.zshrc. If your edits don't take effect, you might be editing the wrong file.
  • Order matters in $PATH: the FIRST matching directory wins. Put ~/bin at the front to override system commands.

Deliverable

Submit:

  1. ~/lab-02/transcript.txt — your command history + ls + PATH dump.
  2. ~/bin/greet — your script (a one-line "I made this" works as a deliverable).
  3. The exact line you added to ~/.bashrc or ~/.zshrc.
  4. A one-paragraph reflection: which compose pattern (pipe vs redirect vs glob) felt most natural, and which surprised you?

Stretch (optional)

  • One-liner challenge: using only one pipeline (no temp files), print the 3 most common login shells on the system.
  • PATH archaeology: for every directory in $PATH, count how many executables it contains. Hint: ls -la <dir> | wc -l looped, or find <dir> -maxdepth 1 -executable.
  • tee: use tee to write a pipeline's output to a file and stdout at the same time. Compose with the error pipeline from task 05.
  • Shebang variants: rewrite greet with #!/bin/sh. Does anything break? Why?
