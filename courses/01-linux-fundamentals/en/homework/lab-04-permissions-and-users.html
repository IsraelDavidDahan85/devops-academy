Lab 04 — Permissions & Users

Stand up two real users, put them in a shared group, and use permissions and special bits to make a workspace that just works. Verify everything with ls -l, id, and stat.

Course01 — Linux Fundamentals
Time~60 minutes
Deliverable~/lab-04/transcript.txt + your /etc/passwd lines for alice & bob + a one-paragraph reflection
Maps to slides34–36 (Part 04 — Operating the system)

Goal

By the end of this lab you can create users, put them in groups, build a shared directory that automatically gives new files the right group via SGID, set chmod modes for three real-world scenarios, and prove the sticky bit on /tmp works.

Setup

You need a real Linux box with sudo access — VM, WSL2, or a cloud instance. Don't use a Docker container, because some of these commands modify /etc/passwd and /etc/shadow and the behaviour inside a container is different from a full system.

mkdir -p ~/lab-04
cd ~/lab-04
script -q transcript.txt        # start recording everything you do
# (run :exit or Ctrl-D when done)
Tip — script(1) records your whole session to transcript.txt. Run all the tasks below inside the recording, then exit.

Tasks

Work through these in order. Read each line before you press Enter.

01Look at the current state

whoami
id
cat /etc/passwd | head -5      # the format: name:x:uid:gid:gecos:home:shell
cat /etc/group | head -5
ls -l /etc/shadow              # only root can read

You should see your own user in /etc/passwd, and a list of groups you belong to from id.

02Create two users

sudo useradd -m -s /bin/bash alice
sudo useradd -m -s /bin/bash bob
sudo passwd alice
sudo passwd bob

id alice
getent passwd alice
ls -ld /home/alice

Verify the users exist, have home directories, and were assigned UIDs (probably 1001 and 1002 on a fresh box).

03Create a shared group

sudo groupadd devs
sudo usermod -aG devs alice
sudo usermod -aG devs bob

getent group devs              # should list both members
id alice                       # devs should be in alice's groups
The -a is critical. usermod -G devs alice (without -a) REPLACES alice's group list with just devs — she loses every other membership, including her primary group's secondaries. Always usermod -aG.

04Make a shared workspace with SGID

sudo mkdir /srv/shared
sudo chown root:devs /srv/shared
sudo chmod 2775 /srv/shared        # leading 2 = SGID
ls -ld /srv/shared                 # mode shows 'rwxrwsr-x' — the 's' is SGID
stat /srv/shared

Now have each user drop a file in:

sudo -u alice touch /srv/shared/from-alice.txt
sudo -u bob   touch /srv/shared/from-bob.txt
ls -l /srv/shared

Both files should show group devs — that's SGID at work. Without SGID, each file would have inherited its creator's primary group instead.

05Three permission scenarios

For each scenario, set the right mode and verify with ls -l.

ScenarioWhat you wantMode
A script anyone can runowner all, others r+x755
A config only root can readowner r+w only600
A log file the devs group can appendowner+group r+w, others r664
 
# Script anyone can run
printf '#!/bin/bash\necho "hi from $USER"\n' | sudo tee /usr/local/bin/hi
sudo chmod 755 /usr/local/bin/hi
sudo -u alice /usr/local/bin/hi

# Config only root can read
sudo touch /etc/important.conf
sudo chmod 600 /etc/important.conf
ls -l /etc/important.conf
sudo -u alice cat /etc/important.conf    # should fail with permission denied

# Shared log writable by devs
sudo touch /srv/shared/devs.log
sudo chown root:devs /srv/shared/devs.log
sudo chmod 664 /srv/shared/devs.log
sudo -u alice sh -c 'echo "alice was here" >> /srv/shared/devs.log'
sudo -u bob   sh -c 'echo "bob was here"   >> /srv/shared/devs.log'
cat /srv/shared/devs.log

06Prove the sticky bit on /tmp

ls -ld /tmp                                # mode ends with 't' — sticky bit
sudo -u alice touch /tmp/alice.txt
sudo -u bob   rm /tmp/alice.txt            # should fail — alice owns it

Without the sticky bit, anyone with write on /tmp could delete anyone else's files. With it, only the owner (or root) can delete.

07Inspect with stat

stat /srv/shared
stat /usr/local/bin/hi
stat /tmp

stat shows the mode in both octal and symbolic form — useful when you can't remember which is which.

08Clean up (optional)

sudo userdel -r alice
sudo userdel -r bob
sudo groupdel devs
sudo rm -rf /srv/shared
sudo rm -f /etc/important.conf /usr/local/bin/hi

Hints

  • After adding a user to a group, they need to log out and back in for the new group to apply to their session. sudo -u alice ... bypasses this because it spawns a fresh session each time.
  • getent passwd alice works even when the user db lives in LDAP or AD, not /etc/passwd. Habit-form it.
  • If chmod 2775 looks weird: the leading 2 is SGID, the 775 is the usual owner/group/other rwx. 4xxx is SUID, 1xxx is sticky.
  • ls -ld shows the directory itself rather than its contents — useful for inspecting modes on the directory.

Deliverable

  3. A one-paragraph reflection: What does the SGID bit on /srv/shared actually buy you? Describe what would go wrong without it.

Stretch

  • Give alice passwordless sudo for one specific command (/usr/local/bin/hi) by dropping a file in /etc/sudoers.d/alice. Test with sudo -n -u alice /usr/local/bin/hi.
  • Use setfacl to grant bob alone (not the whole devs group) write access to a file. Verify with getfacl.
  • Set a default ACL on /srv/shared so that every new file is group-writable by devs regardless of umask: setfacl -d -m g:devs:rw /srv/shared.
