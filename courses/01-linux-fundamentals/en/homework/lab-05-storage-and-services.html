Lab 05 — Storage & Services

Create a fake disk from a regular file, format it, mount it, persist the mount across reboots, then write a systemd service + timer that writes to it on a schedule. Watch the whole thing fire via journalctl.

Course01 — Linux Fundamentals
Time~75 minutes
Deliverable~/lab-05/transcript.txt + your /etc/fstab line + your backup.service & backup.timer + a reflection paragraph
Maps to slides40–43 (Part 04 — Operating the system)

Goal

By the end of this lab you can: build a block device from a regular file (loopback), put a filesystem on it, mount it, make it survive a reboot via /etc/fstab using a UUID (never a device path), and write your own systemd service paired with a timer that runs it on a schedule.

Setup

You need a real Linux box with sudo — VM, WSL2, or a cloud instance. Don't use Docker: loopback mounts and full systemd both behave oddly in containers.

mkdir -p ~/lab-05
cd ~/lab-05
script -q transcript.txt        # start recording everything
# (run :exit or Ctrl-D when done)

Tasks

01Look at what's already mounted

lsblk                          # block devices
findmnt                        # tree of mounts
df -h                          # free space by mount
cat /etc/fstab                 # what mounts at boot

Note the structure: a block device → optional partition → filesystem → mount point.

02Build a loopback device

A loopback device is a regular file the kernel treats as a block device — perfect for practising without a spare disk.

# 256 MB of zeros, then let the kernel attach a loop device to the file
sudo dd if=/dev/zero of=/var/disks/lab.img bs=1M count=256 status=progress
sudo losetup -fP --show /var/disks/lab.img    # prints /dev/loopN (remember it)
lsblk                                          # the new device should show up
What just happened — losetup -fP --show finds the next free loop number (-f), tells the kernel to scan it for partitions (-P), and prints the device path so you can capture it: LOOP=$(sudo losetup -fP --show /var/disks/lab.img).

03Format it with a filesystem

LOOP=$(sudo losetup -j /var/disks/lab.img | awk -F: '{print $1}')
echo "$LOOP"                                   # confirm

sudo mkfs.ext4 -L LAB "$LOOP"                  # ext4 with label LAB
sudo blkid "$LOOP"                             # capture the UUID

Copy the UUID printed by blkid — you'll need it in step 5.

04Mount it manually

sudo mkdir -p /mnt/lab
sudo mount "$LOOP" /mnt/lab
findmnt /mnt/lab                               # verify
df -h /mnt/lab                                 # see the free space

# write something
echo "hello from $(date)" | sudo tee /mnt/lab/hello.txt
ls -l /mnt/lab

05Persist the mount via /etc/fstab

A device path like /dev/loop0 can change across reboots — but a UUID never does. Always mount by UUID in /etc/fstab.

UUID=$(sudo blkid -s UUID -o value "$LOOP")
echo "$UUID"

# append the fstab line (verify the exact UUID first!)
echo "UUID=$UUID  /mnt/lab  ext4  loop,defaults,nofail  0  2" | sudo tee -a /etc/fstab

# remount everything from fstab to verify the line works
sudo umount /mnt/lab
sudo mount -a                                  # mount everything in fstab
findmnt /mnt/lab                               # back, mounted via fstab
The nofail option is important. Without it, a missing or broken disk on boot drops the system into emergency-mode rescue shell. nofail says "if this mount fails, keep booting." For non-critical disks (everything except root), always use nofail.

06Write a systemd service

Create /usr/local/bin/lab-tick.sh — the script the service will run:

sudo tee /usr/local/bin/lab-tick.sh >/dev/null <<'EOF'
#!/bin/bash
set -euo pipefail
echo "[$(date --iso-8601=seconds)] tick on $(hostname)" >> /mnt/lab/tick.log
EOF
sudo chmod +x /usr/local/bin/lab-tick.sh
sudo /usr/local/bin/lab-tick.sh                # smoke-test it
cat /mnt/lab/tick.log

Now write the service unit at /etc/systemd/system/lab-tick.service:

sudo tee /etc/systemd/system/lab-tick.service >/dev/null <<'EOF'
[Unit]
Description=Lab 05 — write one tick into the lab disk

[Service]
Type=oneshot
ExecStart=/usr/local/bin/lab-tick.sh
EOF

07Pair it with a timer

And the timer at /etc/systemd/system/lab-tick.timer — same name, .timer extension. systemd auto-links them.

sudo tee /etc/systemd/system/lab-tick.timer >/dev/null <<'EOF'
[Unit]
Description=Run lab-tick every minute

[Timer]
OnBootSec=30s
OnUnitActiveSec=1min
AccuracySec=1s
Persistent=true

[Install]
WantedBy=timers.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable --now lab-tick.timer
systemctl list-timers --all | grep lab-tick     # see when it next fires

08Watch it fire

journalctl -u lab-tick.service -f               # follow it live; Ctrl-C to exit
# wait ~2 minutes
cat /mnt/lab/tick.log                           # entries every minute

09Inspect the unit and recent activity

systemctl status lab-tick.service
systemctl status lab-tick.timer
systemctl cat lab-tick.service                  # show the unit file the running config came from
journalctl -u lab-tick.service --since "10 min ago"

10Clean up (optional but recommended)

sudo systemctl disable --now lab-tick.timer
sudo rm /etc/systemd/system/lab-tick.{service,timer}
sudo systemctl daemon-reload

sudo umount /mnt/lab
sudo sed -i.bak "/$UUID/d" /etc/fstab           # remove the fstab line
sudo losetup -d "$LOOP"
sudo rm /var/disks/lab.img /usr/local/bin/lab-tick.sh
sudo rmdir /mnt/lab

Hints

  • Always mount by UUID (or LABEL) in /etc/fstab, never by device path. Device numbers shift between boots.
  • If mount -a fails, your boot will fail too. Test it before rebooting — that's the whole point of running it manually.
  • systemctl daemon-reload is required after editing any unit file. Forget it and you'll see your old config silently in effect.
  • A timer file's name must match its service file — foo.timer triggers foo.service by default. To target a different service, add Unit=other.service under [Timer].
  • Persistent=true tells systemd: if the timer would have fired while the machine was off, run it once at boot to catch up.

Deliverable

Stretch

  • Change the timer to use a calendar expression (OnCalendar=*:0/5 — every 5 minutes on the wall clock) instead of OnUnitActiveSec. Verify with systemd-analyze calendar "*:0/5".
  • Add RandomizedDelaySec=10s under [Timer] and observe the actual fire times scatter.
  • Resize the loopback: stop the timer, truncate -s 512M /var/disks/lab.img, losetup -c "$LOOP", resize2fs "$LOOP", remount, verify df -h /mnt/lab shows the new size.
  • Make the service drop the entry under a non-root user. Add User=youruser to [Service] and chown the mount accordingly.
