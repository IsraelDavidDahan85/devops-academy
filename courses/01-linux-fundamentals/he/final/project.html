פרויקט גמר — להפוך מכונת לינוקס לשלכם

להקים מכונת לינוקס חדשה, להתאים אותה מקצה לקצה, ולתעד מה קורה מתחת. הפרויקט שמחבר את כל המעבדות בקורס.

קורס01 — יסודות לינוקס
זמן3–5 שעות (פרוס על כמה ישיבות)
תוצרריפו Git (או .tar.gz) הכולל: README.md, boot.md, system-report.txt, users.md, storage.md, dotfiles/, bin/, systemd/
פורמטדחיפה ל-GitHub (מועדף) — הגישו את ה-URL של הריפו — או ארכיון כקובץ.

היכולת שמוכחת

אתם יכולים לשבת מול מכונת לינוקס חדשה ולא מוכרת ובתוך שעה להפוך אותה למתאימה לדרך שלכם — עם prompt קריא, aliases שמתאימים להרגלים שלכם, $PATH עובד שכולל סקריפטים משלכם, והבנה ברורה של מה המכונה באמת עשתה בין הדלקה ל-login.

זה ההבדל בין "אני יודע להריץ פקודות" ל"המכונה הזו שלי."

הברירה

נמסרת לכם מכונת לינוקס חדשה. היא עולה ישר ל-shell עם ברירות מחדל, צבעי ברירת מחדל, בלי aliases, בלי סקריפטים מותאמים, בלי סיפור. המשימה שלכם: להפוך אותה לשלכם, ולתעד גם מה עשיתם וגם איך המכונה נראתה מתחת.

זה בדיוק מה שכל מהנדס/ת עושה במחשב נייד חדש או בכל פעם שמקימים שרת. תרגול מודע של זה פעם אחת מאיץ את כל הקריירה שלכם על לינוקס.

דרישות

1 · להקים מכונת לינוקס אמיתית

כל אחד מאלה מתאים:

  • VM מקומי (multipass, UTM, VirtualBox, Lima, VMware) שמריץ Ubuntu, Debian, Fedora, Arch או דומה — כל דבר עם systemd.
  • VM בענן (DigitalOcean droplet, AWS EC2, GCP Compute, Linode, Hetzner — אפילו free tier).
  • WSL2 ב-Windows (עובד, אבל סקציית ה-boot תהיה חריגה — תרשמו את זה ב-boot.md).

זה חייב להיות טרי. אל תשתמשו ב-dotfiles הקיימים שלכם או במכונה שאתם משתמשים בה יומיומית.

2 · להתאים את סביבת ה-shell

בקובץ rc של ה-shell שלכם (~/.bashrc או ~/.zshrc), הקימו:

  • לפחות 3 aliases שאתם באמת תשתמשו בהם יומיומית. דוגמאות: ll='ls -lah', gs='git status', ..='cd ..'. לא להעתיק רשימות גנריות — בחרו כאלה שמתאימים לאיך שאתם חושבים.
  • PS1 מותאם שמציג לפחות שם משתמש, hostname, ותיקיית עבודה. בונוס: צבע, ה-git branch הנוכחי.
  • $PATH מעודכן שכולל ~/bin (בהתחלה, כדי שהסקריפטים שלכם ינצחו פקודות מערכת בשם זהה).

Aliases / PS1 / PATH חייבים לשרוד shell חדש — המבחן: סגרו את הטרמינל, פתחו מחדש, ה-alias עדיין עובד בלי להריץ שום דבר.

3 · לכתוב לפחות סקריפט אחד ב-~/bin

כל סקריפט שתשתמשו בו באמת. דוגמאות:

  • up — מריץ apt update && apt upgrade (או המקבילה בדיסטרו שלכם) עם sudo.
  • weather — curl wttr.in/Tel-Aviv או דומה.
  • backup-notes — מריץ rsync של ~/notes לתיקיית backup עם timestamp.
  • greet — הסקריפט ממעבדה 02, נשמר.

הסקריפט חייב להיות executable (chmod +x), עם shebang (#!/usr/bin/env bash), ולרוץ מכל תיקייה (מוכיח שה-$PATH מוגדר נכון).

4 · לכתוב boot.md — סיפור ה-boot של המכונה

תעדו את רצף ה-boot כפי שנצפה ב-VM שלכם. השתמשו בתבנית של מעבדה 03 כנקודת התחלה, אבל כל ערך חייב להיות מהמכונה שלכם.

לכסות לפחות:

  • Firmware/bootloader שהדיסטרו משתמש בו (BIOS/UEFI, GRUB/systemd-boot).
  • גרסת קרנל (מתוך uname -r) ו-5 השורות הראשונות של dmesg.
  • ה-systemd target ברירת המחדל במכונה שלכם, ועוד 3 שירותים שהתחילו ב-boot (השתמשו ב-systemctl list-units --type=service).
  • דוגמה אחת לכל אחד מ-7 סוגי הקבצים (-, d, l, c, b, p, s) — פלט מלא של ls -l.
  • שלוש תיקיות FHS שפתחתם באמת, עם הערה אחת לכל תיקייה ("/etc במכונה זו מכיל 142 קבצים. המעניין ביותר היה…").

5 · לבנות system-report.txt עם pipes

קובץ אחד. שנוצר על ידי לפחות שלוש קומפוזיציות שונות עם pipes, הפניות או globs. דוגמה:

{
  echo "=== 10 הקבצים הגדולים ב-/var/log ==="
  ls -lS /var/log 2>/dev/null | head -n 11

  echo "=== מספר חבילות מותקנות ==="
  dpkg -l 2>/dev/null | grep -c "^ii" || rpm -qa 2>/dev/null | wc -l

  echo "=== משתמשים עם login shells ==="
  cat /etc/passwd | grep -v "/nologin\|/false" | cut -d: -f1
} > system-report.txt

המטרה אינה ה-pipelines המדויקים — אלא שהרכבתם אותם בעצמכם והקובץ מוכיח שאתם מבינים את אינסטלציית ה-shell.

6 · הוסיפו משתמש לא-root עם קבוצה משותפת

המכונה שקיבלתם הופעלה כ-root או כמשתמש ברירת מחדל של הענן. אל תפעלו כ-root לטווח הארוך. צרו משתמש אמיתי לעצמכם, פלוס קבוצה משותפת, והשתמשו בהרשאות הנכונות על תיקייה משותפת:

  • צרו משתמש חדש (השם שלכם) עם תיקיית בית ו-shell אמיתי.
  • הוסיפו אותו ל-sudo (או wheel ב-RHEL) כדי שיוכל להריץ פקודות פריבילגיות.
  • צרו קבוצה devs, הוסיפו את עצמכם, וגרמו ל-/srv/shared להיות בבעלות root:devs במצב 2775 (SGID — ראו מעבדה 04).
  • אמתו עם id, getent, ו-ls -ld /srv/shared.

תעדו את הפקודות המדויקות ב-users.md, כולל העתק של שורות /etc/passwd ו-/etc/group שהוספתם.

7 · מאונטו volume נוסף

הוסיפו אחסון מעבר ל-filesystem הראשי וגרמו לו לשרוד reboot:

  • אפשר לחבר דיסק שני (UI של הענן / ה-VM host) או ליצור loopback device מקובץ רגיל (ראו מעבדה 05).
  • פרמטו אותו עם ext4 (או xfs), תנו לו label, ומאונטו אותו ב-/data.
  • שמרו את ה-mount ב-/etc/fstab לפי UUID (לעולם לא device path) עם nofail.
  • הריצו reboot. אמתו שה-mount שורד — בדקו עם findmnt /data.

תעדו את ה-layout (פלט lsblk לפני ואחרי) ואת שורת ה-/etc/fstab שלכם ב-storage.md.

8 · פרסו שירות systemd מותאם

החליפו את הריפלקס של cron בזוג אמיתי של .service + .timer (ראו מעבדה 05):

  • כתבו סקריפט קטן תחת /usr/local/bin/ שעושה משהו שימושי — מוסיף heartbeat ל-/data/heartbeat.log, שומר snapshot של df -h, או מסובב את דוח המערכת שלכם.
  • שמרו קובץ .service תחת /etc/systemd/system/ שמריץ את הסקריפט כמשתמש הלא-root שלכם.
  • שמרו קובץ .timer מותאם שמפעיל אותו לפי לוח זמנים (כל 5 דקות זה מספיק לדמו).
  • systemctl daemon-reload, enable --now את ה-timer, אז בדקו systemctl list-timers ו-journalctl -u <svc> -f.

הוסיפו את שני קבצי ה-unit לריפו תחת systemd/.

9 · הגישו כריפו Git

אתחלו Git ב-VM. שימו את התוצרים תחת מבנה הגיוני:

my-linux-final/
├── README.md              # פתיחה של דף אחד: מה כאן, איך לקרוא
├── boot.md                # נרטיב ה-boot שלכם
├── users.md               # איך הקמתם את המשתמש, הקבוצה, /srv/shared
├── storage.md             # מבנה הדיסקים + שורת ה-fstab שלכם
├── system-report.txt      # snapshot מערכת שנבנה ב-pipes
├── dotfiles/
│   ├── bashrc-snippet     # השורות שהוספתם ל-~/.bashrc
│   └── ps1-example.txt    # screenshot או העתק של ה-prompt
├── bin/
│   └── greet              # (או הסקריפט האחר שלכם)
└── systemd/
    ├── heartbeat.service  # ה-service unit שלכם
    └── heartbeat.timer    # ה-timer המותאם

דחפו ל-GitHub והגישו את ה-URL של הריפו.

קריטריוני הצלחה

עברתם כשבודק/ת יכול/ה:

  1. לקרוא את ה-README.md שלכם ולהבין את הסקופ של מה עשיתם תוך פחות משתי דקות.
  2. לוודא שה-aliases שלכם אמיתיים (להעתיק לתוך shell משלהם — הם עובדים).
  3. לקרוא boot.md ולמצוא ערכים קונקרטיים וספציפיים למכונה (לא placeholders של תבנית) לכל שלב ב-boot, סוגי הקבצים, והערות ה-FHS.
  4. לקרוא system-report.txt ולומר איזה pipe / redirect / glob הפיק כל סקציה.
  5. להסתכל על הסקריפט ב-bin/, להריץ עליו shellcheck, ולא לראות אזהרות קריטיות.
  6. לקרוא users.md ולאמת שיצרתם משתמש אמיתי לא-root עם sudo + תיקייה משותפת /srv/shared עם SGID.
  7. לקרוא storage.md, לראות שורת fstab מבוססת UUID, ולאמת שה-volume הנוסף מאונטם ב-/data אחרי reboot.
  8. להעתיק את קבצי ה-unit מתיקיית systemd/ שלכם למכונה אחרת, להריץ daemon-reload, להפעיל את ה-timer, ולראות אותו רץ מקצה לקצה דרך journalctl.

מה לא מודדים

  • אסתטיקה של ה-prompt. שיעבוד בשבילכם.
  • איזה דיסטרו בחרתם.
  • כמה aliases מעבר לשלושה הנדרשים — שלושה אמיתיים מנצחים שלושים שהועתקו.
  • אם השתמשתם ב-bash או zsh.

אתגרים (אופציונלי, אך מומלץ)

  • שני dotfiles נוספים: הוסיפו ~/.gitconfig עם שם + email שלכם, ו-~/.tmux.conf אם אתם משתמשים ב-tmux.
  • סקריפט התקנה: כתבו setup.sh בריפו שלכם, שכשמריצים אותו על VM חדש, מקים את הכל על ידי symlinks של dotfiles והוספת סקריפטים ל-~/bin. אידמפוטנטי.
  • One-liner לבריאות שירותים: הוסיפו ל-system-report.txt את מספר ה-units שנכשלו (systemctl --failed), והסבירו ב-boot.md מה כל כשל אומר במכונה שלכם.
  • פרסום פומבי של הריפו: הפכו אותו לפומבי. הסטודנט/ית הבא/ה בתוכנית יכול/ה ללמוד מההגדרה שלכם.
